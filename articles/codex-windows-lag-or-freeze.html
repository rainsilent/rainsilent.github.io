Codex Windows 版本用一陣子後很 Lag，還會卡住

在超級大量使用 Codex Windows 版本一段時間後，某一個當下它就突然變得很慢。

一開始只是新對話送不出去。點送出後一直卡在轉圈圈，甚至重新再開一個新對話也是一樣卡著。後來連載入舊的對話都變得超慢，整個 App 看起來不是普通 lag，而是卡在某個地方過不去。

我一開始也先嘗試最直覺的處理方式：重開整個 App、重新安裝、把不要的對話都 Archive。結果都沒有太大幫助，新對話一樣卡得很嚴重。

原本我還用 ChatGPT 幫我找網路上的經驗，但那些建議對我的狀況幫助不大。甚至 AI 還叫我把整個 .codex 目錄改掉，結果就是我的對話都消失，一切都要重來。

所以後來我乾脆進到 .codex 目錄裡，用 Codex CLI 請它直接檢查本機狀態。下面是它協助我做的一些處理方式。雖然最後 CPU 還是衝得有點高，但至少對話可以送出去了。

這篇不是官方教學，只是我的使用 Codex 幫我處理問題的紀錄，僅供參考，因為這問題應該還是得靠 OpenAI 來優化這個應用了吧...

ChatGPT 一開始還往程式權限問題的方向判斷，但我後來覺得不太像。真的卡權限的話，通常會看到 approval、allow、permission 這類很明確的提示。可是我這次是連輸入送出都還沒真正跑起來，就已經卡在轉圈圈了。

可以先檢查哪裡

Codex App 在 Windows 上會使用這個資料夾：

%USERPROFILE%\.codex

裡面會放一些本機狀態、歷史紀錄、暫存資料、設定檔等等。換句話說，Codex App 很多「看起來像是雲端服務慢」的狀況，其實也可能跟本機這包資料有關。

我後來主要看的是這幾個地方：

  • logs_2.sqlite*
  • sessions
  • archived_sessions
  • .tmp\plugins
  • plugin 設定

這些不一定每個人都會出問題，但如果 App 已經卡到不能正常使用，至少可以先從這幾個方向下手，不要一開始就整包 .codex 改名或刪掉。

第一個可以調整的地方：log 資料

第一個先看 logs_2.sqlite。如果這個檔案變得很大，App 啟動或載入時有可能會被拖慢。

我的做法不是直接刪除，而是先關掉 Codex App，然後把這些檔案移到備份資料夾：

logs_2.sqlite
logs_2.sqlite-wal
logs_2.sqlite-shm

重開後，Codex App 會自己重新建立新的 log 檔。

這一步對我有一點幫助，但不是完整解法。也就是說，log 太大可能是原因之一，但不一定是唯一原因。

第二個可以調整的地方：archived conversations

接著我看了 archived_sessions。這裡放的是已封存的對話紀錄。

如果 archived conversations 很多，App 在載入或整理歷史時可能會變慢。我的做法一樣不是刪除，而是先把整個 archived_sessions 移到備份資料夾，再讓 Codex App 建一個新的空資料夾。

這一步對我比較有感。移走後，App 的反應有比較正常一點。

代價是 App 裡暫時看不到原本封存的對話。不過資料還在備份裡，需要時可以再搬回來。而且反正都 Archive 了...就真的有需要再說吧!

第三個可以調整的地方：plugin 暫存和不常用 plugin

.tmp\plugins 比較像 plugin marketplace 或 catalog 的暫存資料。清掉後 App 會再重建，所以這個相對沒有那麼可怕。

這一步對我來說不是最大改善來源，比較像是順手清一下，幫忙排除是不是暫存資料在拖慢。

另外，如果平常不常用文件、試算表、簡報、PDF 這類 plugin，也可以先暫時停用，看看啟動和使用時會不會好一點。

這不是說 plugin 一定有問題，而是當 App 已經卡住時，先減少它需要載入的東西，會比較容易判斷問題到底卡在哪裡。

不建議一開始就刪的東西

.codex 裡有些檔案看起來也很可疑，我本來看到也覺得亂七八糟很阿雜，但 Codex 說不要亂刪，例如：

auth.json
config.toml
state_5.sqlite
memories_1.sqlite
goals_1.sqlite
plugins
skills

這些可能和登入、設定、記憶、狀態或工具有關。刪了不一定會變快，反而可能讓 App 狀態更亂，到時候就又要重來一次。

總之呢，希望 OpenAI 快點處理一下囉，這優化也太差了吧!!!
