我把 AI Agent 做成一個 Agentic 開發團隊

這次不是單純叫 AI 幫我寫 code。

我原本只是想整理一個程式開發專案裡的幾個 routine，像是 QA、code review/security、負責專案輪巡的 agent，還有定期推進開發的 RD agent。看起來每個 routine 都有在跑，但真的放在一起看，就會發現有些任務重複，有些角色邊界不清楚，一輪能推進的事情不多，進度變得很慢；同時記憶也散在不同地方，有些是長期偏好，有些是 bot 的工作日誌，有些又是專案狀態。

更麻煩的是，原本把 agent 的能力定義直接放在專案資料夾裡。短期看起來方便，但久了就會變成 PM、RD、QA、Security 這些職能都被綁死在單一專案，下一個專案想重用時，還要重新拆 prompt、拆規則、拆記憶。表面上是自動化，實際上是很多 agent 在同一個系統裡互相踩線，而且能力沒有辦法乾淨搬走。

後來我覺得真正要解的不是「多加幾個 prompt」，而是把它當成一個 AI 開發團隊來設計。

第一個關鍵決定，是把「角色能力」跟「專案記憶」拆開。

我後來用一個很簡單的分層，shared_agents/ 放可重用的角色能力，例如 PM Bot、RD Bot、QA Bot、Security；project_workspace/<project>/ 放某個專案自己的狀態、報告、記憶和輸出。

大概像這樣。

workspace/
  shared_agents/
    pm/
    rd/
    qa/
    security/
  project_workspace/
    minijira/
      pm/
      rd/
      qa/
      security/
      memory/

這個拆法很重要，因為如果 QA Bot 的規則裡混了某個專案特定的 CVE、false positive、API endpoint，以後它就不再是一個可重用的 QA，而只是那個專案的一段 prompt。

第二個關鍵決定，是不要讓一個 agent 做所有事。

這點其實也跟我查到的參考資料很接近。

MAGIS 把 GitHub issue resolution 拆成 Manager、Repository Custodian、Developer、QA Engineer 四種角色。在 SWE-bench 實驗中，MAGIS 解決了 13.94% 的 GitHub issues，解決率比直接使用 GPT-4 高 8 倍。

來源：MAGIS: LLM-Based Multi-Agent Framework for GitHub Issue Resolution

TheBotCompany 把 continuous software development 拆成 Strategy、Execution、Verification 三個階段，並保留非同步的人類監督。它的重點不是一次性完成小任務，而是讓 agent team 能持續推進開發，並透過 verification phase 抓出原本會留下來的 defect。

來源：Self-Organizing Multi-Agent Systems for Continuous Software Development

TestingXperts 在 QA automation 的文章裡提到，傳統測試自動化常卡在維護成本，QA team 可能花到 50% 的時間維護既有測試；multi-agent pipeline 則可以讓 source control、test design、execution、environment、reporting 各自負責一段流程。

來源：How Multi-Agent Systems Are Transforming QA Automation

換成我自己的理解，分開做事的好處是責任邊界比較清楚，prompt 不會越寫越肥，失敗時比較知道是哪一段出問題，也比較容易把 PM、RD、QA、Security 這些能力拿去別的專案重用。代價也很明顯，流程設計會變複雜，agent 之間要有明確交接格式，記憶和狀態不能亂放，不然只是把一個大混亂拆成很多小混亂。

最後角色切成這樣。

角色負責什麼不負責什麼
PM Bot寫 SPEC、推進 assignee、管理 gate、更新 PRD不寫 code，不自己 approve
RD Bot寫 code、做 EVT、push feature branch、開 PR不直接 merge main
QA Botcode review、功能測試、regression、merge/reject PR不自己寫功能、不自己判斷 CVE 嚴重度
Security靜態安全分析、CVE、威脅模型、寫 security report不跑動態測試、不直接開單
CodexPR 靜態 review，給建議不取代 QA merge gate

這裡有一個我覺得很經典的問題，到底誰 code review？誰 commit code？

一開始我也差點手癢再拆一個 Code Reviewer Bot。後來想想不對，這東西很可能只是多一個轉述 Codex 意見的人，沒有真的增加品質。最後我定下來的是 RD Bot 負責 commit code，QA Bot 負責 code review 和 merge。因為 code review 本來就是驗收的一部分，要看的是「這個改動有沒有符合 Story」、「邏輯和邊界條件有沒有問題」、「功能測試有沒有過」。

不過 QA Bot 本身也不能是一個模糊的大包角色。它可以是一個排程，但規則裡要把子職能拆清楚。

QA 子職能做什麼
QA Reviewer看 PR diff、code quality、acceptance criteria
Test Designer根據 spec 產測試案例
Test Runner實際跑 pytest、regression 或 e2e
Environment Checker確認環境、DB、service、dependency 是否正常
Report Writer整理 QA result、issue comment、review 結論

這樣做的原因很簡單，QA 要負責 code review、功能測試、regression、安全驗證、開單和回報，如果全部混在一段規則裡，很快又會變成另一個什麼都做、什麼都說不清楚的 agent。

所以流程變成這樣。

PM Bot
  -> 寫規格、整理 Story、需要人類決策時通知我

RD Bot
  -> 讀 Story
  -> 寫 code
  -> 做 EVT
  -> push feature branch
  -> 開 PR
  -> 留言 @codex 請 Codex review

Codex
  -> 在 PR 上做靜態 review
  -> 留建議

RD Bot 下一輪
  -> 讀 Codex comment
  -> 有需要就修正並 push 到同一個 branch

QA Bot
  -> 檢查 PR evidence
  -> code review
  -> functional test / regression
  -> 通過才 merge main

Security
  -> 獨立排程掃 main branch
  -> 寫 findings/report
  -> QA Bot 讀 report 後做動態驗證與開單

這套裡面我最在意的是「不能只相信 AI 說它做完了」。

因為 agent 很常給你一個看起來很完整的完成回報，但真的去看 repo，可能測試沒跑、branch 沒推，甚至只是改了半套。它不是故意騙你，但如果流程沒有要求 evidence，人就很容易被那個「看起來很完整」的回報騙過去。

所以 RD Bot 每個 PR 都必須附 evidence。

EVT result:
Changed files:
Test command:
Test output path:
Risk note:
Rollback plan:

QA Bot merge 前不是只看 diff，而是要看 evidence。這件事很像真的工程流程，不是「我改好了」就可以進 main，而是要留下可檢查的證據。

另一個重點是 Security Gate。Security 平常跟 RD/QA 平行跑，不會卡低風險開發。但如果 PR 碰到 auth、permission、dependency、secret、DB migration、file upload、external API、logging、user data、token、session 這些範圍，就會觸發 security checklist。High/Critical 擋 merge，Medium 要 PM approval，Low 可以轉 backlog。

例如只是改 UI wording，不需要 Security 介入。但如果 RD Bot 動到 token、session、permission check，那就不能只靠 QA 跑功能測試。畫面看起來正常，不代表權限邏輯沒有破。這樣 Security 不會變成每天擋路的流程，但真正高風險的改動也不會被當成一般功能測試混過去。

這次也踩到一個很實際的坑，RD Bot 明明應該開 PR，結果還是 merge 回 main。原本以為是舊排程，後來確認是新 SKILL.md 還不夠具體，因為環境沒有 gh CLI，所以開 PR 步驟失敗後行為退回舊流程。最後改成用 GitHub API 開 PR，也讓 QA Bot 用 GitHub API 查 open PR，不再依賴本地 branch 狀態。

這個坑其實很有價值，它提醒我 agent 流程不能只寫「開 PR」，要寫清楚「用什麼方法開 PR、失敗時不能 merge main、應該怎麼回報」。AI 開發團隊不是 prompt 寫得漂亮就好，它需要像真正 SOP 一樣，把錯誤路徑也設計進去。

同樣的道理，失敗也不能只是一筆 log 或一張 bug。失敗要變成結構化記憶，不然 agent 很容易一直踩同一個坑。

這點也是參考資料裡反覆出現的方向。

TheBotCompany 把 Verification 放成獨立階段，目的就是抓出那些如果沒有驗證就會留下來的 defect。

來源：Self-Organizing Multi-Agent Systems for Continuous Software Development

TestingXperts 提到 multi-agent system 的持續性不只來自分工，也來自 agents 能根據過去結果持續調整邏輯；它把這種能力視為 MAS 的核心價值之一。

來源：How Multi-Agent Systems Are Transforming QA Automation

Bot-Driven Development 也把 bot-driven workflow 跟 faster feedback、continuous learning、efficiency 連在一起。換句話說，bot 不應該只是一直執行任務，也要能從執行結果裡學到下次不要怎麼做。

來源：Bot-Driven Development

所以我把失敗記憶拆成幾類。

記憶用途
failed_attempts.md記錄 EVT 或測試失敗
anti_patterns.md記錄重複踩坑的錯誤模式
known_regressions.md記錄已知 regression 與避免方式
lessons_learned.md記錄複雜修復後真正學到的東西

這不是只有 QA 要寫。RD Bot 開發前也要讀 anti_patterns.md 和 known_regressions.md，EVT 失敗也要寫 failed_attempts.md。不然同樣的錯誤，只是從 QA 那邊變成 RD 那邊重複發生。

還有一個原則，遇到需要人類決策的事情，要明確通知，不要默默卡住。

AI agent 可以自動執行，但不能假裝它有產品決策權。需求取捨、風險接受、優先順序，這些還是要回到人類。所以 PM Bot 寫好規格後，要到 Heptabase 日記提醒我 review；遇到需要人類決定的事情，也要用固定方式寫到 Heptabase，而不是留在 log 裡等我自己發現。

最後還加了一個很有趣的 feedback loop，RD Bot 開 PR 後自動留言 @codex，Codex review 後由 RD Bot 下一輪讀 comment 修正，再交給 QA Bot。這樣 QA 看到的 PR 不是第一版，而是已經經過一次靜態 review 和修正的版本。

但這套也還沒完美。現在我還沒完全解掉幾件事，像是 agent 寫入記憶的粒度很難抓；Heptabase 提醒如果太多，最後也會變噪音；QA Bot 到底要不要有權限直接 merge，我其實也還在觀察。

跑完這一輪後，我比較有感的是，AI 會不會寫 code 反而不是最大問題。真正麻煩的是，它寫完之後誰檢查、誰擋下來、誰記住這次失敗，下次不要再犯。

所以這次對我來說，比較不像是在測一個 prompt，而是在測一套小型開發流程。只是隊員剛好都是 agent，而且每個都很需要被管。
