GitHub Pages 的 sitemap 一直被 Google Search Console 顯示「無法讀取」？先不用改了，問題不在你身上

如果你的網站放在 GitHub Pages 預設網域，也就是 username.github.io，然後 Google Search Console 一直顯示 sitemap「無法讀取」，這件事很可能不是你的 sitemap 寫錯。

你可能已經確認過：

  • sitemap.xml 格式正確
  • robots.txt 沒有擋到
  • sitemap 用驗證工具檢查正常
  • 甚至已經用 AI 找過問題，但還是一直無法讓 GSC 檢查通過

那問題可能就在這裡：不是你的檔案有問題，而是 Google 讀 GitHub Pages 預設網域上的 sitemap 時，本來就可能會卡住。

目前比較像是 Googlebot 在讀取 github.io 上的 sitemap 時會失敗，錯誤可能會是 URL_FETCH_STATUS_MISC_ERROR。這個狀況從 2025 年開始就有人在 GitHub 社群回報，到 2026 年還沒有穩定的官方修正。

比較能證明問題不在 sitemap 本身的是，同一份 sitemap：

  • 丟到 Bing Webmaster Tools，可以正常讀
  • GitHub Pages 綁上 custom domain 後，再送 Google，也可以正常讀

所以不用一直改 .nojekyll、robots.txt、sitemap 檔名，或一直重送測試檔。這些方法通常不是解這個問題的。

那可以怎麼做咧？

第一種，就是直接用 Google Search Console 的「網址審查」送首頁。

如果網站頁面不多，這其實最省事。Google 只要爬到首頁，就會順著首頁上的連結去找其他文章。之後新增文章，再單獨送那篇文章的網址就好。

第二種，是綁 custom domain，或乾脆換 hosting。

綁自訂網域後，再重新在 GSC 建立資源並提交 sitemap，目前看起來是比較能根治的方式。Netlify、Vercel、Cloudflare Pages 這類服務，也有使用者實測 sitemap 可以被 Google 正常讀取。

但如果你跟我一樣很懶，不想買網域、不想搬站、不想重設部署，也不想再跟 GSC 的 sitemap 狀態耗下去，那最實際的做法就是：直接去提交首頁。

sitemap 讀不到不代表網站不能被 Google 找到。只要首頁能被索引，文章連結也放在頁面上，Google 還是可以順著首頁把其他內容找出來。對頁面數量不多的個人站來說，這通常已經夠用了。
